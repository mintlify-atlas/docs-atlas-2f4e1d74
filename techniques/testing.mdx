NestJS is designed with testability in mind. The `@nestjs/testing` package provides `Test.createTestingModule()` — a lightweight version of the NestJS module system that lets you assemble a testing context with the same DI wiring as production code, while still allowing you to swap out real dependencies for mocks.

## Installation

`@nestjs/testing` ships with every new NestJS project generated by the CLI. If you need to install it manually:

```bash
npm install -D @nestjs/testing
```

NestJS projects use [Jest](https://jestjs.io/) by default. Test files follow the `*.spec.ts` convention for unit tests and `*.e2e-spec.ts` for end-to-end tests.

## Unit testing a service

Unit tests isolate a single class (typically a service) and replace its dependencies with mocks or stubs.

```typescript cats/cats.service.spec.ts
import { Test } from '@nestjs/testing';
import { CatsService } from './cats.service';
import { Cat } from './interfaces/cat.interface';

describe('CatsService', () => {
  let catsService: CatsService;

  beforeEach(async () => {
    const moduleRef = await Test.createTestingModule({
      providers: [CatsService],
    }).compile();

    catsService = moduleRef.get<CatsService>(CatsService);
  });

  describe('findAll', () => {
    it('should return an array of cats', async () => {
      const result: Cat[] = [{ name: 'Pixel', age: 2, breed: 'Bombay' }];
      // access private property for seeding test data
      (catsService as any).cats = result;

      await expect(catsService.findAll()).resolves.toBe(result);
    });
  });

  describe('create', () => {
    it('should add a new cat', () => {
      const cat: Cat = { name: 'Mochi', age: 1, breed: 'Siamese' };
      catsService.create(cat);
      expect((catsService as any).cats).toContainEqual(cat);
    });
  });
});
```

`Test.createTestingModule()` accepts the same `ModuleMetadata` as `@Module()` — `imports`, `controllers`, `providers`. Call `.compile()` to instantiate all providers and then use `.get()` to retrieve instances.

## Unit testing a controller

```typescript cats/cats.controller.spec.ts
import { Test } from '@nestjs/testing';
import { CatsController } from './cats.controller';
import { CatsService } from './cats.service';

describe('CatsController', () => {
  let catsController: CatsController;
  let catsService: CatsService;

  beforeEach(async () => {
    const moduleRef = await Test.createTestingModule({
      controllers: [CatsController],
      providers: [CatsService],
    }).compile();

    catsController = moduleRef.get<CatsController>(CatsController);
    catsService = moduleRef.get<CatsService>(CatsService);
  });

  it('should return all cats', async () => {
    const cats = [{ name: 'Pixel', age: 2, breed: 'Bombay' }];
    jest.spyOn(catsService, 'findAll').mockResolvedValue(cats);

    expect(await catsController.findAll()).toBe(cats);
  });
});
```

## Mocking providers

Replace a real provider with a mock by chaining `.overrideProvider().useValue()` before calling `.compile()`:

```typescript cats/cats.service.spec.ts
const mockCatsService = {
  findAll: jest.fn().mockResolvedValue([]),
  create: jest.fn(),
};

const moduleRef = await Test.createTestingModule({
  controllers: [CatsController],
  providers: [CatsService],
})
  .overrideProvider(CatsService)
  .useValue(mockCatsService)
  .compile();
```

You can also override guards, interceptors, filters, and pipes:

```typescript
moduleRef = await Test.createTestingModule({ ... })
  .overrideGuard(AuthGuard)
  .useValue({ canActivate: () => true })
  .overridePipe(ValidationPipe)
  .useValue({ transform: (value) => value })
  .compile();
```

### Auto-mocking with `useMocker`

For large modules with many providers, use `useMocker` to automatically generate Jest mock objects for every unresolved dependency:

```typescript
import { createMock } from '@golevelup/ts-jest';

const moduleRef = await Test.createTestingModule({
  controllers: [CatsController],
})
  .useMocker(createMock)
  .compile();
```

## End-to-end (e2e) testing

E2e tests spin up the full application and send real HTTP requests using [Supertest](https://github.com/ladjs/supertest).

```bash
npm install -D supertest @types/supertest
```

```typescript e2e/cats/cats.e2e-spec.ts
import { INestApplication } from '@nestjs/common';
import { Test } from '@nestjs/testing';
import * as request from 'supertest';
import { CatsModule } from '../../src/cats/cats.module';
import { CatsService } from '../../src/cats/cats.service';
import { CoreModule } from '../../src/core/core.module';

describe('Cats (e2e)', () => {
  const catsService = { findAll: () => ['testCat'] };

  let app: INestApplication;

  beforeAll(async () => {
    const moduleRef = await Test.createTestingModule({
      imports: [CatsModule, CoreModule],
    })
      .overrideProvider(CatsService)
      .useValue(catsService)
      .compile();

    app = moduleRef.createNestApplication();
    await app.init();
  });

  it('GET /cats returns 200', () => {
    return request(app.getHttpServer())
      .get('/cats')
      .expect(200)
      .expect({ data: catsService.findAll() });
  });

  afterAll(async () => {
    await app.close();
  });
});
```

Key steps in an e2e test:
1. Create the testing module the same way as a unit test.
2. Call `moduleRef.createNestApplication()` to get an `INestApplication` instance.
3. Call `app.init()` to bootstrap the application (runs lifecycle hooks, applies global pipes/guards, etc.).
4. Use `app.getHttpServer()` as the Supertest target.
5. Call `app.close()` in `afterAll` to release resources.

<Tip>
Apply your global pipes, guards, and filters inside the test setup to match production behavior:

```typescript
app = moduleRef.createNestApplication();
app.useGlobalPipes(new ValidationPipe({ whitelist: true }));
await app.init();
```
</Tip>

## Overriding entire modules

Replace a feature module wholesale using `overrideModule()`. This is useful for replacing a database module with an in-memory version:

```typescript
const moduleRef = await Test.createTestingModule({
  imports: [AppModule],
})
  .overrideModule(DatabaseModule)
  .useModule(InMemoryDatabaseModule)
  .compile();
```

## TestingModuleBuilder API

| Method | Description |
| ------ | ----------- |
| `overrideProvider(token)` | Replace a provider. Chain with `.useValue()`, `.useClass()`, or `.useFactory()`. |
| `overrideGuard(guard)` | Replace a guard. |
| `overrideInterceptor(interceptor)` | Replace an interceptor. |
| `overrideFilter(filter)` | Replace an exception filter. |
| `overridePipe(pipe)` | Replace a pipe. |
| `overrideModule(module)` | Replace an entire module. Chain with `.useModule()`. |
| `useMocker(factory)` | Auto-mock all unresolved providers using the provided factory. |
| `setLogger(logger)` | Use a custom logger during test compilation. |
| `compile()` | Compile the module and return a `TestingModule`. |

## Test file conventions

- Unit test files live alongside the source file they test: `cats.service.ts` → `cats.service.spec.ts`
- E2e test files live in an `e2e/` or `test/` directory at the project root
- Jest is configured in `jest.config.ts` or the `jest` field in `package.json`
- Use `--testPathPattern` to run a specific spec file: `npx jest cats.service`
